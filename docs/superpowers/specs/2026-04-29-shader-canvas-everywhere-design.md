# Shader canvas hero on every page

## Goal

Replace the per-page SVG hero image (currently rendered above blog post bodies and a handful of other pages) with the existing `HeroCanvas` shader-canvas component, currently only present on the home page. Every page on the site gets an animated text-shader hero whose phrases are derived from that page's content. Existing inline SVGs in post bodies are unaffected.

## Context

`HeroCanvas` (`src/components/svelte/HeroCanvas.svelte`) is a Svelte component that renders an animated typing field of short phrases through a WebGL shader. It already accepts `phrases: string[]` as a prop and is loaded with `client:only="svelte"` from `HomeLayout.astro`. The component handles `prefers-reduced-motion`, `visibilitychange` pause/resume, and runs at a deliberate 12 FPS to keep CPU/battery cost low.

`PostLayout` renders `<img class="post-hero" src={image} alt="" />` and `PageLayout` renders `<img class="page-hero" src={image} alt="" />` above page content, where `image` is a frontmatter field pointing to a hand-authored SVG in `public/assets/images/posts/`. The same `image` value is passed to `BaseLayout` for use as og:image. `TalkLayout` and `GigLayout` do not currently render hero images. `HomeLayout` already renders its own `HeroCanvas` instance inline (loaded via `client:only="svelte"`).

## Architecture overview

A single `HeroCanvas` instance lives at the top of `BaseLayout.astro`, marked `transition:persist="hero-canvas"`. Astro's view transitions keep the same DOM element and Svelte instance across navigations rather than tearing down and remounting on each page.

Each page exposes its phrases to the persistent canvas via a `<meta name="hero-phrases" content="phrase1|phrase2|...">` tag rendered into `<head>`. `HeroCanvas` reads this on mount and on every `astro:page-load` event, replacing its internal `activePhrases` pool. Existing rows finish typing their current phrase, hit idle, then pick from the new pool — producing a natural cross-fade via the canvas's existing trail decay. No new transition logic is needed.

The meta tag is the sole source of phrases. `HeroCanvas`'s `phrases` prop is removed in favour of reading directly from the DOM, so the persisted component instance always picks up the latest page's phrases regardless of how Astro hydration interacts with `transition:persist`.

The reduced-motion path replaces the canvas with a plain `<img>` pointing at the per-page static image (with fallback to a shared default). This is lighter than running WebGL at all, accessibility-friendlier, and identical to what social previews show.

Per-page static images double as og:image. They live under `src/assets/heroes/` and go through Astro's image pipeline (`getImage()`) for optimisation, format negotiation, and cache-busting. They're generated by a manual script that uses `agent-browser` to screenshot the live canvas for each page.

## Component changes

### HeroCanvas

Three changes to `src/components/svelte/HeroCanvas.svelte`:

1. Phrase source. Remove the `phrases` prop. Add an internal `readPhrasesFromMeta()` helper that reads `document.querySelector('meta[name="hero-phrases"]')`, splits on `|`, and trims. Call it on mount and on every `astro:page-load` event to refresh `activePhrases`.
2. Navigation listener. On mount, register an `astro:page-load` listener that calls `readPhrasesFromMeta()` and replaces `activePhrases`. The existing typing simulation handles the visual transition naturally — current rows finish, idle, then pick from the new pool.
3. Reduced-motion path. Replace the current "simulate ~60 frames then stop" block with: if `prefers-reduced-motion: reduce`, render `<img>` with a fallback image URL (read from a `<meta name="hero-image-url">` tag) instead of the two `<canvas>` elements. The shader, font, and burst logic only initialise when motion is allowed.

The shader source, phrase typing logic, burst behaviour, and 12 FPS loop stay unchanged.

### BaseLayout

Renders the persistent `HeroCanvas` above `<main>`:

```astro
<HeroCanvas client:only="svelte" transition:persist="hero-canvas" />
```

The component takes no props — it reads everything from meta tags. `BaseLayout` renders both:

```astro
<meta name="hero-phrases" content={phrases.join("|")} />
<meta name="hero-image-url" content={heroImageUrl} />
```

Where `heroImageUrl` is the resolved URL string returned by `getImage({ src: heroImage })`. Passing a string (rather than the `ImageMetadata` object) avoids serialisation issues with `client:only` Svelte components.

`BaseLayout` accepts new `phrases: string[]` and `heroImage: ImageMetadata` props. Every layout (`PostLayout`, `PageLayout`, `TalkLayout`, `GigLayout`, `HomeLayout`) computes its phrase set and looks up its hero image, then passes both through.

### Per-layout phrase assembly

A site-wide baseline lives in `src/data/hero.ts`:

```ts
export const SITE_BASELINE_PHRASES = ["benswift", "Ben Swift", "Human-Scale AI"];
```

`PostLayout` assembles phrases as `[...SITE_BASELINE_PHRASES, ...frontmatter.tags, ...frontmatter.heroPhrases]`, where `heroPhrases` is populated by the remark plugin (below).

`PageLayout` assembles `[...SITE_BASELINE_PHRASES, ...(frontmatter.heroPhrases ?? []), title]`. `TalkLayout` and `GigLayout` (whose source files are pure markdown without much structure) use `[...SITE_BASELINE_PHRASES, title, event]` (skipping null fields).

`HomeLayout` removes its inline `<HeroCanvas client:only="svelte" phrases={heroPhrases} />` (now handled by `BaseLayout`) and instead passes the existing curated list combined with all blog post titles to `BaseLayout` as the `phrases` prop.

## Remark plugin for content phrases

A new plugin `src/plugins/remark-hero-phrases.ts` runs during MDX/MD parsing:

1. Walk the AST looking for `heading` nodes (depth 2-4) and `emphasis`/`strong` nodes.
2. Extract their text content, trim, normalise whitespace.
3. Filter to length ≤ 36 characters (matches `MAX_PHRASE_LEN` in `HeroCanvas`).
4. De-duplicate case-insensitively while preserving original casing.
5. Write the resulting array to `data.astro.frontmatter.heroPhrases`.

The plugin is registered in `astro.config.mjs` under `markdown.remarkPlugins` (and likewise for the MDX integration's `remarkPlugins`).

The blog collection schema in `src/content.config.ts` adds an optional `heroPhrases: z.array(z.string()).optional()` field so the typed frontmatter exposes the value to layouts.

## Per-page static images

### Storage and lookup

Images live at `src/assets/heroes/<slug>.png`, where `<slug>` is the date-prefixed slug produced by `pathToRkey()` in `scripts/lib/posts.ts` (e.g. `2026-04-28-agentic-coding-studio`). This is the same identifier used as the atproto rkey, so the lookup module, the generation script, and the atproto integration all derive slugs the same way. A shared default lives at `src/assets/og-default.png`.

Note: this changes the naming convention from the existing SVGs (which used the terminal slug only, e.g. `agentic-coding-studio.svg`). The new date-prefixed form is collision-free. The existing SVGs are deleted as part of the cleanup, so no migration is needed.

A small lookup module `src/utils/hero-image.ts`:

```ts
import type { ImageMetadata } from "astro";

const heroImages = import.meta.glob<{ default: ImageMetadata }>(
  "/src/assets/heroes/*.{png,jpg}",
  { eager: true, import: "default" },
);

import defaultHero from "../assets/og-default.png";

export function getHeroImage(slug: string): ImageMetadata {
  const key = `/src/assets/heroes/${slug}.png`;
  return heroImages[key]?.default ?? defaultHero;
}
```

Layouts call `getHeroImage(slug)` and pass the resulting `ImageMetadata` to `BaseLayout`. `BaseLayout` calls `getImage({ src: heroImage, format: "png" })` once to produce a single resolved URL string used for both the og:image meta tag and the `<meta name="hero-image-url">` tag (consumed by HeroCanvas under reduced motion). PNG is chosen for broad social-media scraper compatibility.

For non-blog pages (research, teaching, gigs, talks), the layout passes a stable identifier or simply omits the slug, falling back to the default image. Per-page generated images are blog-only in this iteration.

### Generation script

`scripts/generate-hero-images.ts`:

1. Pre-condition: `pnpm build` has been run so `dist/` exists. The script either fails fast if `dist/` is missing or invokes the build itself; preference is to fail fast and document the prerequisite in the npm script.
2. Read `atproto-state.json` to get the baseline `contentHashes` map.
3. Run `discoverPosts(blogDir)` (re-using `scripts/lib/posts.ts`) to enumerate current posts and compute fresh content hashes.
4. Build a work list: posts whose path is absent from `oldHashes` OR whose hash differs from `oldHashes[post.path]`. (New posts and edited posts both get regenerated.)
5. Spin up `pnpm preview` in a child process. Wait for the server to be ready (poll `/` for HTTP 200).
6. For each post in the work list: invoke `agent-browser` to navigate to the post URL, wait ~3 seconds for the canvas to settle, screenshot the `.hero-canvas` region (the wrapping div in `HeroCanvas.svelte`), save to `src/assets/heroes/<slug>.png`. Use the same slug derivation as the layout (matching `entry.id`).
7. Tear down the preview server.
8. Print a summary: how many regenerated, how many skipped, and how many remain on the default fallback.

A separate command regenerates only the default fallback:

`scripts/generate-default-hero.ts` opens the home page (which has the densest phrase pool), screenshots, saves to `src/assets/og-default.png`. Run when the canvas styling changes substantially.

Both surfaced as `pnpm gen:hero-images` and `pnpm gen:default-hero` in `package.json`.

No CI integration in this iteration. The user runs scripts locally before pushing meaningful content changes.

## Performance notes

- One WebGL context per browser session (not per page) thanks to `transition:persist`. Better than per-page teardown.
- 12 FPS loop, paused on `visibilitychange` when the tab is hidden, paused entirely under `prefers-reduced-motion`.
- Reduced-motion users get a static `<img>` — no WebGL initialisation at all.
- Bundle: `shaderpad` and Svelte runtime now ship to every visitor (currently only home-page visitors). Estimated 10-30KB extra over the existing baseline.
- og:image and reduced-motion fallback are routed through Astro's image pipeline, so they're optimised, fingerprinted, and cache-friendly.
